上周我搭了一个个人博客，整个过程几乎没手写 HTML。用了三个不同的 AI 工具分工协作，从设计稿到线上部署，总共花了不到两小时。

这篇文章不是教程，更像是一次复盘——记录这个流程是怎么跑通的，踩了什么坑，以及为什么我觉得这种工作方式值得尝试。

## 第一步：用 AI 生成设计稿

一切从一个叫 `ui-ux-pro-skill` 的工具开始。

这是一个专门做 UI 设计的 AI skill。我给了它几个关键词：个人博客、暗色主题、Bento Grid 布局、红色强调色。它输出了一整页完整的 HTML——包括 CSS 变量定义、字体搭配（Instrument Serif + Outfit）、响应式断点、甚至微交互的 hover 动画。

说实话，它的设计水准超出我的预期。暗色背景上点缀微弱的红色径向渐变，文章卡片用 Bento Grid 排列，第一篇是大卡片，其他是小卡片。没有那种"AI 味"很重的通用模板感，反而有点像精心设计过的作品集页面。

这一步的价值在于：**AI 把设计决策和代码实现合并成了一个动作。** 以前我得先找参考、选配色、画 wireframe，再慢慢写 CSS。现在一句 prompt 搞定。

但出来的是一个**静态 HTML 文件**——所有内容都写死在里面。文章标题是假的，统计数据是硬编码的（"47 篇文章"），没有后台，不能更新。

所以我需要把它变成一个**真正的博客系统**。

## 第二步：用 Gemini 把设计稿变成博客

我做了一件可能有点反直觉的事：把设计稿丢给了另一个 AI。

具体来说，我打开了 Gemini，把整个 `landing-page.html` 的代码贴进去，然后说：

> "这是一个落地页的设计稿，帮我改造成一个 GitHub Pages 博客。需要：Python 构建脚本、Markdown 文章管理、GitHub Actions 自动部署。"

Gemini 的输出让我印象深刻。它做的事情包括：

- **拆解设计稿**：把内联 CSS 提取成独立的 `style.css`
- **创建模板系统**：用 HTML 注释占位符（`<!-- POST_TITLE -->`、`<!-- POSTS_PLACEHOLDER -->`）实现动态内容注入
- **写构建脚本**：`builder.py` 用 Python 读取 Markdown 文件（带 frontmatter），解析后填入模板生成静态 HTML
- **配 CI/CD**：`deploy.yml` 用 GitHub Actions 在 push 时自动构建并部署到 `gh-pages` 分支

这里有个有意思的点：**Gemini 理解了我的设计意图**。它没有把暗色主题改掉，保留了 Bento Grid 布局，甚至正确地给第一篇文章加了 `featured` 类（让它占两列宽）。它不是在"重新生成"，而是在"理解后重组"。

当然，输出不是完美的。几个问题：

- 首页文章计数显示为 "20" 而不是 "2"（模板占位符旁边有个残留的 `0`）
- "12 主题"是写死的，不是动态计算的
- 没有标签页、RSS、搜索等功能
- Footer 的 Twitter、GitHub、RSS 链接都是 `#`（假链接）

但这些是**可以迭代的问题**，不是架构问题。框架搭好了，剩下的就是填充和打磨。

## 第三步：用 Agent 持续迭代

这一步回到了 OpenClaw——我日常使用的 AI agent。

和前两步不同，这一步不是一次性生成，而是**持续对话、逐步改进**：

**修 bug**：首页的 `<!-- POST_COUNT -->0` 问题。Gemini 在占位符后面留了个 `0` 作为 fallback，结果替换后显示成 `20`。这个 bug 很典型——AI 生成代码时喜欢加 fallback，但模板替换的逻辑和 fallback 文字会冲突。

**加功能**：我让 agent 帮我加了标签页。它在 `builder.py` 里新增了 `build_tags()` 函数，创建了 `tags.html` 模板，写了标签药丸和分组列表的 CSS。过程中它自己判断了"标签锚点用 `-` 连接"、"按文章数量降序排列"这些细节。

**加 RSS**：我说了一句"加上 RSS"，它就自动生成了 RSS 2.0 XML，包含了 `atom:link` 自引用、RFC 2822 日期格式这些标准细节，还顺手修了 footer 的死链接。

**写 Skill 文档**：最后它把整个博客的维护经验整理成了一份可复用的 Skill 文档——包括架构概览、发布流程、常见坑、快速参考命令。

这一步的核心价值是：**AI agent 记住了上下文，并且能做出合理的工程决策。** 它不是在执行"把这个 div 改成那个颜色"，而是在理解"这个博客需要什么功能"后自主实施。

## 这个流程说明了什么？

回头来看，三个 AI 工具各司其职：

| 工具 | 角色 | 输入 | 输出 |
|------|------|------|------|
| ui-ux-pro-skill | 设计师 | 文字描述 | 完整设计稿 |
| Gemini | 架构师 | 设计稿 + 需求 | 博客工程结构 |
| OpenClaw Agent | 工程师 | 修改需求 | 功能迭代 + bug 修复 |

这不是"AI 帮我写了个网页"这么简单。更准确地说，是 **AI 承担了每个环节中重复性最高的部分，而我把精力花在了决策上**——选什么风格、要不要标签页、RSS 有没有必要。

我做的事情是：提需求、审输出、做取舍。AI 做的事情是：理解需求、生成代码、处理细节。

## 踩坑总结

1. **AI 生成的代码会有"装饰性冗余"**：比如占位符旁边的 fallback 数字、用不到的 CSS 变量。清理成本不高，但得看一眼。
2. **不同 AI 的风格差异很大**：ui-ux-pro-skill 倾向炫技（微交互、渐变），Gemini 倾向保守（标准结构、注释详细），agent 倾向实用（能跑就行，细节后补）。了解这些差异有助于选择合适的工具。
3. **上下文传递是关键**：Gemini 能做好是因为我给了完整的设计稿，agent 能做好是因为整个项目结构已经在了。AI 的输出质量，很大程度上取决于你给的上下文质量。
4. **不要期望一步到位**：设计稿要改、结构要调、bug 要修。把"一次性完美"的期望改成"快速出稿 + 持续迭代"，体验会好很多。
