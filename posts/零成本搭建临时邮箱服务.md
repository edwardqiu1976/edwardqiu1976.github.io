# 零成本搭建临时邮箱服务：一次有趣的实践

**日期**：2026-04-01  
**技术栈**：Cloudflare Workers、Temp-Mail API

---

## 为什么要自己搭建临时邮箱？

注册一个网站账号，网站要求填写邮箱——这大概是每个网民都逃不过的场景。

用真实邮箱？后续等待你的，是源源不断的营销邮件、验证码轰炸、还有不知道什么时候就被泄露到暗网的风险。

不用？抱歉，大部分网站根本不允许跳过这一步。

临时邮箱就是为这个场景诞生的。市面上存在不少临时邮箱服务，但体验过的朋友都知道：要么几分钟就过期，要么广告满屏飞，要么写着写着突然就挂了。你永远不知道它什么时候会突然抛弃你。

作为一个能自己托管在 Cloudflare Workers 上的开发者，我合计了一下：Cloudflare Workers 本身免费且稳定，Temp-Mail 有现成的 API，自己做一个似乎也不是什么难题。

说干就干。

---

## 过程与踩坑

### 第一步：研究 API

临时邮箱的原理并不复杂：生成一个随机邮箱地址代人收信，所有发到这个地址的邮件会被转发到一个管理后台，用户随时可以查看。

先把 Temp-Mail API 的文档翻了一遍，了解了域名配置、邮件获取、删除等核心接口。心里大概有底了。

### 第二步：封装成 skill

既然以后可能还会用到，不如一次性封装成一个可复用的 skill。下次再用时，几分钟就能部署上线，不用从头折腾。

这个 skill 包含了 Cloudflare Workers 脚手架、Temp-Mail API 集成、以及管理员后台的基本功能。

### 第三步：第一次部署踩坑

兴冲冲部署上线，分配了一个测试域名，信心满满。

测试发邮件——

**0 封。**

排查了半小时，终于定位到问题：域名配置错误。不是所有域名都适合接收邮件，选错了域名，整个服务就是摆设。

这大概是这类服务最容易被忽视的一个点：你以为配置好了，其实邮件压根到不了。

### 第四步：换域名重试

这回选了一个验证过能收邮件的域名，重新部署。

用管理员账号登录查看——竟然发现有历史邮件。看来这服务之前有人用过，一直在线跑着。

发一封测试邮件，这次顺利收到了。

---

## 一些体会

这次实践下来，有几点感受：

1. **域名配置是生死线**——看似简单的配置，选错了就完全用不了
2. **小工具很有用**——临时邮箱这种看似不起眼的东西，实际场景中能省很多麻烦
3. **自动化会让人上瘾**——把部署流程封装成 skill 之后，下次再用只需要几分钟，这种感觉用过就回不去了
4. **测试真的很重要**——不要自信到跳过测试步骤（说的就是我）
