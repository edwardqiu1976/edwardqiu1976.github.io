零成本域名管理：从注册到自动化的全流程实践

免费注册子域名、免费解析托管、免费自动化续期提醒——踩坑记录全公开。

为什么折腾这个

我一直想整理自己的域名——不花钱那种。

买域名不贵，.com 一年也就几十块。但问题是：你买了一个之后，会想买第二个、第三个。等回过神来，域名池里躺着七八个，有的已经忘了当初为什么要注册，续费时又舍不得放弃。年复一年，这笔钱就默默流走了。

这次我决定换个思路：找免费的。

免费域名服务不少，但大多数要么注册完就没人管了，要么管理界面反人类。直到我发现了一个叫 DNSHE 的服务——它提供免费子域名，根域名覆盖好几个区域，有效期最长能到 10 年。

配合 Cloudflare 的免费 DNS 解析，正好搭一套完整的域名管理体系。

整个过程从下午搞到晚上，踩了不少坑，但最终效果还挺满意的。下面把完整流程拆开讲。

第一步：理解架构

先说清楚整体方案，不然中间容易绕晕。

核心思路是 双层架构：

| 层级 | 工具 | 职责 | |------|------|------| | 注册层 | DNSHE | 子域名注册、续期、生命周期管理 | | 解析层 | Cloudflare | DNS 解析、CDN 加速、安全防护 |

DNSHE 管"拥有"，Cloudflare 管"对外服务"。各司其职，互不干扰。

这有什么好处？以后如果想换解析服务商，DNSHE 那边不用动；DNSHE 那边的免费政策变了，Cloudflare 的解析记录也不受影响。解耦就是自由。

第二步：Cloudflare API 配置

很多人用 Cloudflare 只是为了 CDN 加速，但它的 DNS 管理 API 才是真正值得深挖的能力。要通过脚本自动化操作，第一步是创建 API Token。

登录 Cloudflare Dashboard → My Profile → API Tokens → Create Custom Token。权限按需分配：

| 权限 | 用途 | |------|------| | Zone:Read | 读取域名信息 | | DNS:Edit | 管理 DNS 记录 | | Workers:Edit | 管理 Workers（后续扩展用） | | D1/KV/R2 读写 | 边缘存储操作 |

拿到 Token 后，一个 curl 验证连通性：

bash curl -s -H "Authorization: Bearer $CF_API_TOKEN" \ "https://api.cloudflare.com/client/v4/zones" | jq '.result[].name'

如果返回了你的域名列表，说明 Token 没问题。Cloudflare 的 DNS 解析是免费的，支持 A/AAAA/CNAME/MX/TXT 全部常见记录类型，还有 DNSSEC、DNS 防火墙等高级功能。对个人项目来说，这就是免费的天花板了。

第三步：DNSHE 子域名注册

它是什么

DNSHE 提供免费子域名注册，根域名覆盖全球多个区域：

  • ccwu.cc — 华人常用
  • us.ci — 美属维尔京群岛
  • cn.mt — 马耳他
  • cn.bi — 布隆迪

有效期分 360 天和 3600 天两档，到期可以续期。3600 天就是将近 10 年，注册一次基本不用担心短期过期。

API 连通踩坑

DNSHE 提供 API Key + API Secret 的双密钥认证。我踩的第一个坑：一开始只给了 Key 没给 Secret，请求直接 401 失败。补上 Secret 后才正常：

bash curl -s "https://api.dnshe.com/v1/subdomains&action=list" \ -H "X-Api-Key: $DNSHE_API_KEY" \ -H "X-Api-Secret: $DNSHE_API_SECRET"

这个教训很朴素：双密钥认证的服务，两个都得给。单给一个不会报"缺少 Secret"，而是直接 401，容易误导你怀疑别的地方出了问题。

注册流程

注册新子域名的流程不复杂：先查可用根域名，再提交注册请求，最后配置 DNS 解析。但手动操作太原始，后面我会讲怎么用脚本自动化。

第四步：从手动到自动化

光有 API 还不够，得把常用操作封装成可复用的工具。我把 DNSHE 的管理功能整理成四个模块：

| 功能模块 | 说明 | |----------|------| | 自动续期 | 检查到期时间 + 单个/批量续期 | | DNS 记录管理 | A/AAAA/CNAME/MX/TXT 增删改查 | | 注册新子域名 | 检查可用根域名 + 注册 + 自动配 DNS | | Cloudflare 联动 | DNSHE ↔ Cloudflare 同步脚本 |

每个模块的实现其实不复杂，核心就是 curl + jq。但把这些逻辑沉淀成文档和脚本之后，以后再操作就不需要翻 API 文档了。

第五步：密钥安全——别把钥匙挂门上

这一节单独拎出来讲，因为这可能是整个实践中最重要的教训。

问题

最初我把 API Key 和 Token 明文写在了一个工作文档里——这个文件每次对话都会被加载到 AI 的上下文里，等于把钥匙挂在门上，谁都能看见。

解决方案

采用 .env + .gitignore 的经典方案：

  1. 新建 .env：集中存放所有密钥 bash CF_API_TOKEN=your_token_here CF_ACCOUNT_ID=your_account_id_here DNSHE_API_KEY=your_key_here DNSHE_API_SECRET=your_secret_here

  2. 新建 .gitignore：排除 .env，防止误提交到 git

  3. 所有文档中的密钥替换为环境变量引用：$CF_API_TOKEN、$DNSHE_API_KEY 等

  4. 全盘扫描清理残留：确保没有遗漏的明文 key

经验总结

API 密钥不应该出现在任何会被加载到 AI 对话上下文的文件中。.env + .gitignore 是最轻量且有效的方案——密钥只存在于本地文件，不进 git，不进 AI 上下文。

以后用任何需要 API Key 的服务，第一时间就做这个安全设置。别等出了问题再补。

第六步：续期提醒——别让域名悄悄过期

免费域名最大的风险就是忘了续期。

我的处理方式是设了两个定时提醒：有效期短的域名（360 天）在到期前 180 天提醒，有效期长的（3600 天）也按提前 180 天的标准来。到时间了自动推送通知，不用自己惦记。

360 天的域名差不多注册后半年就要开始关注了，节奏还挺紧凑的。

复盘：踩了哪些坑

| 坑 | 教训 | |----|------| | API Key 只给了一个，没给 Secret | 双密钥认证两个都要，别漏 | | 密钥明文写进文档 | .env + .gitignore 是标准姿势 | | DNSHE API 偶尔 502 | 免费服务不稳定，需要容错设计 | | 没有搞清部署流程就操作 | 先看文档和架构，再动手 |

总结

整套方案的成本是 零。

| 项目 | 费用 | |------|------| | DNSHE 子域名注册 | 免费 | | Cloudflare DNS 解析 | 免费 | | 自动化脚本（自己写） | 免费 | | 续期提醒（Cron） | 免费 |

四个关键经验：

  1. 双层架构：注册和解析分开，各司其职，降低耦合
  2. 密钥安全：.env + .gitignore，别把钥匙挂门上
  3. 自动化优先：能用脚本解决的就别手动记
  4. 免费的天花板：Cloudflare 免费 DNS + DNSHE 免费子域名 = 零成本域名管理

如果你也有几个域名在手里吃灰，不妨试试这个思路。零成本试错，最坏的结果也不过是浪费一个下午。
